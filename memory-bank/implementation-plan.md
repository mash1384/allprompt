# LLM 프롬프트 헬퍼: 상세 구현 계획 (v1.3)

## 목표:
제공된 프로젝트 계획서(v1.3), 기술 스택, 코딩 규칙을 바탕으로 "LLM 프롬프트용 코드 스니펫 생성 도우미" 데스크톱 애플리케이션을 구현합니다. 이 계획은 AI 개발자(Claude 등)가 따라 할 수 있도록 단계별 지침과 각 단계의 검증 방법을 명시합니다. 코드 자체는 포함하지 않습니다.

**참고 문서:**
*   프로젝트 계획서 v1.3 (이전 대화의 최종 버전)
*   기술 스택 요약 (이전 대화의 최종 버전)
*   코딩 규칙 (`.cursor/rules/rules.mdc` 내용)

---
**중요**
**구현 방법:**

*   **PySide/PyQt 스타일시트 (QSS):** 이 단계들의 핵심 구현 수단입니다. 별도의 `.qss` 파일을 만들어 스타일을 관리하고, 애플리케이션 시작 시 로드하여 적용하는 방식을 권장합니다.
*   **아이콘 리소스:** SVG 형식을 우선적으로 고려하고, Qt의 리소스 시스템(`.qrc`)을 사용하여 아이콘을 애플리케이션에 효율적으로 내장합니다.
# LLM 프롬프트 헬퍼: 상세 구현 계획 - UI/UX 개선 단계 (상세 설명)

## 목표:
애플리케이션의 전반적인 시각적 디자인과 사용자 경험을 현대적이고 미니멀한 다크 모드 기반의 전문 개발 도구 스타일로 개선합니다. 일관된 아이콘, 명확한 레이아웃, 가독성 높은 색상 및 폰트 적용을 목표로 합니다.

---

*   **단계 1: 디자인 컨셉 및 스타일 가이드 정의**
    *   **지침:**
        1.  **주요 디자인 원칙 수립:** "미니멀리즘", "명확성", "일관성", "전문성"을 주요 디자인 원칙으로 설정합니다.
        2.  **색상 팔레트 정의 (다크 모드 기반):**
            *   **배경색:** 매우 어두운 회색 (예: `#20232A`, `#282C34`)을 기본 배경으로 사용합니다.
            *   **패널/영역 구분 배경색:** 기본 배경보다 약간 밝거나 어두운 회색 (예: `#1A1D23`, `#32363F`)을 사용하여 미묘한 영역 구분을 합니다.
            *   **텍스트 색상:** 밝은 회색 또는 거의 흰색 (예: `#ABB2BF`, `#DCE0E8`)을 사용하여 가독성을 확보합니다.
            *   **강조/선택 색상:** 밝은 파란색 계열 (예: `#528BFF`, `#61AFEF`) 또는 밝은 회색을 사용하여 선택된 항목, 활성 버튼 등을 표시합니다.
            *   **경고/오류 색상:** 상황에 맞는 색상 (예: 노란색 계열 경고, 붉은색 계열 오류)을 일관되게 사용합니다.
        3.  **폰트 선택:** 시스템 기본 산세리프 폰트 중 가독성이 뛰어난 것(예: macOS의 SF Pro, Windows의 Segoe UI, Linux의 Noto Sans 등)을 우선 사용하거나, Inter, Roboto 등 웹 폰트 중 적절한 것을 선택하여 일관되게 적용합니다. 기본 텍스트 크기와 제목, 부제목 등 계층에 따른 크기 변화 규칙을 정합니다.
        4.  **아이콘 스타일 정의:** 단색의 단순하고 명확한 라인 아이콘 스타일을 채택합니다. 아이콘의 크기는 일관되게 유지하고(예: 16x16 또는 24x24 픽셀), 색상은 주변 텍스트 색상 또는 강조 색상과 조화를 이루도록 합니다. (예: Material Symbols의 Outlined 스타일, Feather Icons 등 참고)
        5.  **기본 레이아웃 및 간격 규칙:** 컴포넌트 간 기본 여백(예: 8px, 12px, 16px), 내부 패딩 값 등을 정의하여 시각적 리듬감과 일관성을 부여합니다.
    *   **검증:** 정의된 색상 샘플, 폰트 종류 및 크기, 아이콘 스타일 예시, 기본 레이아웃 스케치가 디자인 원칙에 부합하고 명확하게 문서화되었는지 확인합니다.

*   **단계 2: 메인 레이아웃 및 패널 재구성 (2단 구조, 명확한 영역 구분)**
    *   **지침:**
        1.  애플리케이션 창을 명확한 2단 레이아웃으로 구성합니다. 왼쪽은 파일 트리 뷰 패널, 오른쪽은 정보 요약 및 주요 액션 패널로 배치합니다. `QSplitter`를 사용하여 사용자가 패널 크기를 조절할 수 있게 합니다.
        2.  **왼쪽 파일 트리 패널:**
            *   패널 배경색은 스타일 가이드의 "패널 배경색"을 적용합니다.
            *   패널 상단에 "파일/폴더"라는 명확한 헤더 텍스트(또는 아이콘과 함께)를 추가합니다.
            *   패널 하단에는 "숨김 파일/폴더 표시", ".gitignore 규칙 적용" 체크박스를 배치하되, 텍스트 레이블과 체크박스 간 간격을 적절히 조정하고, 전체적인 디자인과 어울리는 스타일(QSS 활용)을 적용합니다.
        3.  **오른쪽 정보/액션 패널:**
            *   패널 배경색은 스타일 가이드의 "패널 배경색" 또는 기본 배경색과 구분되는 색상을 적용합니다.
            *   패널 상단에 "폴더 열기" 버튼을 배치합니다. 버튼 옆이나 아래에 "현재 폴더:" 레이블과 함께 선택된 폴더의 전체 경로를 명확하게 표시합니다.
            *   그 아래 영역에 "선택된 파일: N 개", "총 토큰 수: M 개"와 같은 주요 정보를 그룹화하여 간결하고 명확한 레이블과 값으로 표시합니다. 각 정보 항목 간 충분한 수직 간격을 둡니다.
            *   패널 하단에 "클립보드에 복사" 버튼을 가장 눈에 띄는 주요 액션 버튼으로 크게 배치합니다.
    *   **검증:** 애플리케이션 실행 시 2단 레이아웃이 명확하게 구분되고, 각 패널의 요소들이 논리적이고 시각적으로 균형 있게 배치되었는지 확인합니다. 패널 크기 조절이 부드럽게 동작하는지 확인합니다.

*   **단계 3: 파일 트리 뷰 스타일 개선 (미니멀 아이콘, 명확한 들여쓰기, 깔끔한 상태 표시)**
    *   **지침:**
        1.  **아이콘 적용:** 스타일 가이드에 정의된 라인 아이콘 스타일을 사용하여 폴더(열림/닫힘 상태에 따라 다른 아이콘), 파일 타입별(Python, Markdown 등), 심볼릭 링크, 바이너리 파일 아이콘을 적용합니다. 아이콘 색상은 주변 텍스트 색상과 동일하거나 약간 연하게 처리합니다.
        2.  **들여쓰기 및 확장/축소 아이콘:** 각 트리 레벨별 들여쓰기 값을 명확히 하고(예: 16px 또는 20px), 폴더 확장/축소 아이콘은 단순한 삼각형(예: `►` 또는 `▼`)이나 더하기/빼기 기호 형태의 라인 아이콘을 사용합니다. 트리 연결선은 사용하지 않거나 매우 얇고 연한 색으로 처리하여 깔끔함을 강조합니다.
        3.  **체크박스 스타일:** 기본 OS 스타일 체크박스 대신, QSS를 사용하여 배경색과 체크 표시 모양을 커스텀합니다. (예: 어두운 배경에 밝은 파란색 체크 표시)
        4.  **선택/호버 스타일:** 항목 위로 마우스 커서가 올라갔을 때(호버) 배경색을 미묘하게 변경하고(예: 기본 패널 배경보다 약간 밝은 회색), 항목이 선택되었을 때 배경색을 강조색(예: 스타일 가이드의 밝은 파란색)으로 변경하고 텍스트 색상도 대비되도록 조정합니다.
        5.  **아이템 간 간격:** 각 트리 아이템 간의 수직 간격을 적절히 조절하여 가독성을 높입니다.
    *   **검증:** 트리 뷰가 시각적으로 매우 깔끔하고, 아이콘과 텍스트가 명확하게 구분되며, 선택/호버 상태가 직관적으로 인지되는지 확인합니다. 정보 탐색이 용이해졌는지 확인합니다.

*   **단계 4: 버튼 및 컨트롤 요소 스타일 개선 (플랫 디자인, 명확한 상호작용)**
    *   **지침:**
        1.  **버튼 스타일:** "폴더 열기", "클립보드에 복사" 등의 모든 버튼에 QSS를 적용하여 평평한(flat) 디자인을 기본으로 합니다. 배경색은 패널 배경과 유사하거나 약간 밝게, 텍스트 색상은 가독성 있게 설정합니다. 마우스 호버 시 배경색이나 텍스트 색상이 미묘하게 변경되고, 클릭 시 눌리는 효과를 추가합니다. 주요 액션 버튼("클립보드에 복사")은 다른 버튼보다 시각적으로 더 강조될 수 있도록 색상이나 크기를 조절합니다. 버튼 모서리는 약간 둥글게(예: 3-4px radius) 처리합니다.
        2.  **기타 컨트롤:** 설정 창 등에서 사용될 수 있는 체크박스, 라디오 버튼, 텍스트 입력 필드 등의 스타일도 버튼과 유사한 미니멀하고 일관된 스타일로 QSS를 사용하여 조정합니다.
        3.  **정보 표시 레이블:** "현재 폴더:", "선택된 파일:", "총 토큰 수:" 등의 레이블 텍스트와 실제 값 텍스트의 폰트 크기, 색상, 굵기를 구분하여 정보의 계층을 명확히 합니다. (예: 레이블은 일반 굵기, 값은 약간 더 굵게 또는 다른 색상)
    *   **검증:** 모든 인터랙티브 요소들이 일관된 미니멀한 디자인 스타일을 가지며, 각 요소의 역할과 상태(활성/비활성, 호버, 클릭)가 명확하게 인지되는지 확인합니다.

*   **단계 5: 전체적인 색상 및 폰트 일관성 적용 (QSS 활용)**
    *   **지침:** 애플리케이션의 모든 UI 컴포넌트(메인 창 배경, 각 패널 배경, 텍스트 요소, 구분선, 스크롤바 등)에 스타일 가이드에서 정의된 색상 팔레트와 폰트 설정을 QSS를 통해 일관되게 적용합니다. 구분선은 필요한 경우에만 최소한으로 사용하고, 매우 얇고 연한 색상(예: 패널 배경보다 약간 밝거나 어두운 회색)으로 처리합니다. 스크롤바도 전체적인 다크 테마에 어울리도록 QSS로 스타일을 조정합니다.
    *   **검증:** 애플리케이션 전체적으로 통일된 색상 테마와 폰트 스타일이 적용되어 시각적 일관성과 전문성이 확보되었는지 확인합니다. 가독성에 문제가 없는지, 눈이 편안한지 등을 평가합니다.

*   **단계 6: 상태 표시 및 피드백 디자인 개선 (미니멀하고 명확하게)**
    *   **지침:**
        1.  **진행 상태 표시:** 파일 스캔, 토큰 계산 등 백그라운드 작업 진행 시, 상태 표시줄에 간결한 텍스트 메시지(예: "폴더 분석 중...")와 함께 매우 얇거나 작은 프로그레스 바를 표시하는 것을 고려합니다. 또는, 특정 영역에 미니멀한 스피너 아이콘을 표시할 수도 있습니다. 사용자 인터랙션을 방해하지 않는 선에서 피드백을 제공합니다.
        2.  **알림 메시지 (QMessageBox 등):** 오류, 경고, 성공 알림 메시지 박스의 아이콘, 버튼 스타일, 배경색 등을 애플리케이션 전체 스타일 가이드에 맞춰 QSS로 조정합니다.
        3.  **툴팁:** 툴팁의 배경색과 텍스트 색상도 전체 테마와 일관성을 갖도록 QSS로 조정합니다.
    *   **검증:** 진행 상태 표시와 알림 메시지가 현재 작업 상태를 명확하게 전달하면서도, 전체적인 미니멀한 디자인을 해치지 않고 잘 통합되었는지 확인합니다.

*   **단계 7: 반응형 및 창 크기 조절 시 레이아웃 안정성 확보**
    *   **지침:** 애플리케이션 창 크기를 사용자가 조절할 때, 내부 UI 요소들(패널, 버튼, 텍스트 영역 등)이 깨지지 않고 비율에 맞게 적절히 재배치되거나 크기가 조절되도록 PySide/PyQt의 레이아웃 관리자(`QVBoxLayout`, `QHBoxLayout`, `QGridLayout`, `QSplitter` 등)를 올바르게 사용하고 설정합니다. 최소 창 크기를 지정하여 너무 작게 줄였을 때 UI가 망가지는 것을 방지합니다.
    *   **검증:** 다양한 창 크기 및 비율로 조절해보면서 UI 요소들이 깨지거나 겹치지 않고 안정적으로 유지되며, 모든 정보가 여전히 가독성 있게 표시되는지 확인합니다.

---

**구현 방법:**

*   **PySide/PyQt 스타일시트 (QSS):** 이 단계들의 핵심 구현 수단입니다. 별도의 `.qss` 파일을 만들어 스타일을 관리하고, 애플리케이션 시작 시 로드하여 적용하는 방식을 권장합니다.
*   **아이콘 리소스:** SVG 형식을 우선적으로 고려하고, Qt의 리소스 시스템(`.qrc`)을 사용하여 아이콘을 애플리케이션에 효율적으로 내장합니다.

## Phase 5: 통합 테스트, 최적화 및 배포 준비 (예상: 2주)

**목표:** 애플리케이션의 전체적인 안정성, 성능, 사용성을 검증하고 배포 가능한 형태로 패키징.

*   **단계 28: 종합적인 통합 테스트 (OS별, 다양한 시나리오)**
    *   **지침:** 개발된 모든 기능 조합 다양한 시나리오(대용량 폴더, 복잡한 `.gitignore`, 다양한 파일 타입/크기, 오류 유발, OS별) 테스트.
    *   **검증:** 모든 테스트 시나리오 예상대로 동작, 강제 종료 안 됨, 데이터 정확성, 플랫폼 간 동작 일관성 확인.

*   **단계 29: 성능 프로파일링 및 최적화**
    *   **지침:** `cProfile` 등 사용하여 성능 병목 구간 식별(폴더 스캔, 토큰 계산, UI 업데이트). 코드 최적화.
    *   **검증:** 최적화 후 이전 성능 저하 시나리오 반응 속도/자원 사용량 개선 측정/확인. 일반 사용 환경 쾌적한 성능 제공 주관적 평가.

*   **단계 30: 최종 UI/UX 검토 및 개선 (일관성, 명확성)**
    *   **지침:** 전체 UI/UX 검토. 위젯 배치, 라벨링, 아이콘, 메시지 문구 명확성/일관성 확인. 필요시 툴팁, 도움말 추가. 플랫폼별 UI 가이드라인 참고.
    *   **검증:** 제3자 피드백. UI 직관적, 사용 편리, 전문적 느낌 확인.

*   **단계 31: 애플리케이션 패키징 (PyInstaller, OS별 빌드)**
    *   **지침:** `PyInstaller` 사용하여 Windows/macOS용 독립 실행 파일 생성. `.spec` 파일 수정(아이콘, 의존성, 데이터 파일 등). 각 플랫폼별 빌드/패키징.
    *   **검증:** 패키징 과정 오류 없이 완료 확인. 생성 실행 파일 깨끗한 테스트 환경에서 실행. 패키징 앱 정상 실행, 모든 기능 개발 환경과 동일 동작 철저 검증.

*   **단계 32: 기본 사용자 문서 작성 (README.md)**
    *   **지침:** `README.md` 파일 작성. 애플리케이션 목적, 주요 기능, 설치 방법, 간단한 사용법 가이드, 알려진 문제점/제약 사항 명확/간결 기술.
    *   **검증:** 작성 문서 읽었을 때 처음 사용자 앱 설치/기본 기능 이해/사용 가능한지 확인. 내용 정확/최신 상태 확인.



**지속적인 활동:**

*   **코드 리뷰 및 리팩토링:** 각 단계 완료 후 또는 주기적으로 작성 코드 검토, 코딩 규칙 준수 확인. 필요시 코드 리팩토링.
*   **버전 관리:** 각 의미 있는 변경 사항/단계 완료 시 Git 커밋, 필요시 브랜치 활용 기능 개발.
*   **로깅 확인:** 개발 및 테스트 중 로그 파일을 주기적으로 확인하여 잠재적 문제점 파악.
